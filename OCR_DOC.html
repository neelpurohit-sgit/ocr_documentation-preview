Scoreboard OCR & Smart Capture Pipeline

1. Frame Processing & Throttling (The 150ms Gate)

Objective: Prevent CPU thermal throttling and memory crashes while maintaining real-time responsiveness.

The React Native Vision Camera captures video at 30 to 60 frames per second (FPS). Running Optical Character Recognition (OCR) on 60 frames every second would overload the mobile device. To solve this, the Worklet utilizes a timestamp-based gatekeeper: 

const now = Date.now();
if (now - lastProcessedTime.value < 150) return;
lastProcessedTime.value = now;
  • How it works: The camera ignores all incoming frames until 150 milliseconds have passed since the last successful scan.
  • Result: The OCR engine processes approximately 6 to 7 frames per second. This is fast enough to catch a 1-second clock tick instantly, but slow enough to keep the app running smoothly in the background without draining the battery.

2. Data Extraction via Regex (The AI Brain)

Objective: Transform a raw, unstructured array of OCR text strings into validated, categorized sports data.

When the OCR scans a frame, it returns a "dumb" array of text blocks (e.g., ["ESPN", "LAKERS", "104", "12:30", "4TH", "LIVE"]). The pipeline must intelligently filter out broadcast noise. We achieve this by running four strict Regular Expression (Regex) filters against every word, combined with spatial array validation.

1. The Game Clock: /(\d{1,2})[:.](\d{2})/

This is the primary anchor of the system. It specifically hunts for the time format.

  • How it works: Looks for 1 to 2 digits, followed by a separator, followed by exactly 2 digits.
  • The LED Glitch Fix: Notice it accepts both : and . as separators. Bright LED scoreboards often cause the camera sensor to bleed, erasing the gap in a colon. This ensures the app correctly catches 12.30 as 12:30.
2. The Game Period: /\b(1ST|2ND|3RD|4TH|Q[1-4]|HALF|OT|FINAL|PRD|PERIOD)\b/

Instead of pattern matching, this acts as a strict VIP guest list.

  • How it works: Compares the text against an explicitly defined list of accepted sports periods.
  • The Boundary Rule: The \b (word boundary) at the start and end is critical. Without it, if the TV displayed the word "1STOP", the regex would blindly extract "1ST" from inside it and corrupt our data.
3. Team Scores: /^\d{1,3}$/

Scores are standard numbers, but we must prevent grabbing random broadcast statistics.

  • How it works: The ^ and $ assert the start and end of the string. The text must be a standalone number between 1 and 3 digits (e.g., 7, 14, 112).
  • Conflict Resolution: Before classifying a number as a score, the algorithm checks && !isClock to ensure we do not accidentally count the Game Clock as a team score.
4. Team Names (Regex + Spatial Validation): /^[A-Z]{2,12}$/

This is the most complex extraction, as a team name ("LAKERS") looks mathematically identical to broadcast text ("SPORTS") to a computer.

  • Step 1 (Regex): First, it verifies the word consists only of uppercase letters, between 2 and 12 characters long.
  • Step 2 (Spatial Validation): We apply an isNextToData boolean. The algorithm checks the parsed text array to see if the proposed team name is sitting directly next to a valid Team Score. (e.g., In ["LAKERS", "104"], "LAKERS" is validated as a team. In ["ESPN", "LIVE"], "ESPN" is ignored as noise).

The "Brick Wall" Gate (Ingredient Tallying)

To definitively prove the camera is looking at a scoreboard (and not a microwave clock or a spreadsheet), the pipeline tallies the matched regex ingredients. To unlock the UI and trigger the capture engine, a single frame must possess:

  • At least two Team Scores.
  • Either one Game Period OR one Validated Team Name.

If this threshold is met, the system flags isActuallyScoreboard = true and proceeds to the Proximity Magnet phase.

3. Time & Period Separation (The Proximity Magnet)

Objective: Isolate the exact Game Clock and Period to display a clean UI header (e.g., 1:54 | 2ND), while ignoring false positives like "3rd & 7".

Because football scoreboards often display the "Down and Distance" (e.g., 3rd & 7) near the game clock, the OCR can be easily tricked into grabbing the wrong period.

The Proximity Logic:

  1. Locate the Clock: The algorithm finds the exact array index of the Game Clock.
  2. Filter by Distance: It scans the text array for all words matching the Period regex. Instead of grabbing the first one it sees, it calculates the mathematical distance (array steps) between the Period word and the Clock word.
  3. The Magnet: It permanently locks onto the Period word that is closest to the clock, while explicitly skipping any period word that has an & or AND attached to it.

Resulting UI Output: const headerText = periodWord ? `${clockWord} | ${periodWord}` : clockWord;

4. Smart Capture Engine (Paused vs. Live Games)

Objective: Capture the sharpest possible image immediately, determine the live state of the game, and dynamically trigger single or burst photography.

The capture engine uses a "Pre-Fetch" architecture to ensure high-quality photos before the user's hands shake.

Step 1: The Instant Background Capture

The exact millisecond the Worklet confirms a valid scoreboard, the React UI thread fires captureInitialPhoto(). It silently snaps a high-resolution photo, crops it to the green box coordinates, and stores the URI in a background useRef.

Step 2: The 4-Second Observation Window

Simultaneously, a 4000ms (4-second) setTimeout begins. The Worklet tracks the Game Clock for changes:

  • Normalization: It converts all . to : (turning 12.29 into 12:29) so OCR glitches do not trigger false ticks.
  • The Sticky Tracker: If the clock string changes at all compared to the previous frame, isLiveGame.value permanently flips to true.

Step 3: Resolution & Final Capture

  • Scenario A (Paused Game): If the 4-second timer finishes and the Worklet never detected a clock change, the app assumes a paused screen. It pulls the 1 background photo from the useRef, ignores the burst sequence, and displays the "Paused Game Captured" UI.
  • Scenario B (Live Game): If the Worklet detects a clock change within the 4 seconds, React instantly cancels the timer. It keeps the 1st background photo, calculates the exact hardware processing delay, and snaps 2 more photos exactly 1.0 seconds apart to create a perfect 3-photo sequential burst.
